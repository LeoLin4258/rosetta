# Product

## Register

product

## Users

Rosetta is used by researchers, students, technical writers, developers, founders, and privacy-sensitive business users who need to translate long documents without sending source text to cloud services. They work with papers, books, internal documents, Markdown files, and PDF files, often while checking structure, progress, and export readiness.

## Product Purpose

Rosetta is a local-first desktop workbench for long-form document translation. It connects to a local RWKV translation service or an explicitly configured API, translates document segments in batches, preserves document structure where possible, and keeps imported files, translation cache, and exported results under local user control.

Success means users can import a long document, choose a translation backend, run or retry translation, inspect the result, and export a usable translation without chat workflows, cloud accounts, sync, telemetry, or generic AI features.

## Brand Personality

Calm, private, precise. The interface should feel like a focused desktop tool: quiet hierarchy, clear status, direct controls, and no marketing language inside task surfaces.

## Anti-references

Rosetta should not look or behave like a generic AI assistant, chat product, cloud translation dashboard, collaboration suite, or landing page. Avoid decorative gradients, glass effects, oversized hero treatment, vague AI copy, onboarding theater, and controls that imply upload, login, sync, telemetry, document Q&A, summarization, or rewriting.

## Design Principles

1. Lead with local control: make the active backend, privacy boundary, and file-local workflow obvious.
2. Keep long-document work scannable: favor predictable navigation, compact status, and progressive disclosure.
3. Preserve task focus: expose translation, preview, retry, and export paths without adding generic AI actions.
4. Show operational truth: statuses should state what is ready, blocked, running, or missing in plain language.
5. Protect structure and privacy as product behavior, not decorative claims.

## Accessibility & Inclusion

Use accessible product UI defaults: visible focus states, keyboard-operable controls, sufficient text contrast, semantic labels, reduced-motion-safe transitions, and status copy that does not rely on color alone.
