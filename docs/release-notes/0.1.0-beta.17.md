# Rosetta 0.1.0-beta.17

Rosetta beta.17 focuses on making visual PDF translation more durable,
recoverable, and responsive during long local runs.

## Highlights

- PDF translation now keeps durable per-job and per-language run state, so
  paused, interrupted, or force-quit runs can be recovered instead of leaving
  pages permanently stuck in a transient translating state.
- PDF page translations are stored as canonical per-language page artifacts.
  Opening a PDF task repairs missing metadata, migrates readable older
  artifacts, and resets missing translated pages back to pending.
- PDF deletion is safer for Windows file-lock cases: the job leaves the task
  list first, then cleanup proceeds through a trash directory with a recorded
  follow-up cleanup task if files are still locked.
- Translated PDF panes now show the source page as a backdrop while the
  translated page is loading or still pending, reducing blank page flashes
  during page-by-page translation.
- The PDF worker prewarms the layout model with a synthetic page before it
  reports ready, improving time to first visible translated page on cold runs.
- Local PDF timeline diagnostics record page lifecycle, worker stage timings,
  and aggregate model timings without storing source text, translated text,
  prompts, or model responses.
- PDF paragraph batching was raised for better total-document throughput on
  the local RWKV PDF path.
- Per-job pdf2zh cache handling was tightened so switching or reusing PDF jobs
  does not leak stale worker cache state across documents.

## Supported platforms

- Windows 10/11 x64. The recommended managed runtime uses llama.cpp Vulkan on
  supported AMD, Intel, and NVIDIA GPUs. Machines without a compatible Vulkan
  path may use CPU fallback or an explicitly configured external API.
- macOS 13 or newer on Apple Silicon remains supported through the existing
  local macOS runtime.

Unsigned Windows Preview installers may show a SmartScreen or
unknown-publisher warning on first install unless an Authenticode certificate
is configured during release.

Rosetta remains local-first. Documents, translations, job caches, prompts, and
runtime logs are not uploaded by the release or update system.
